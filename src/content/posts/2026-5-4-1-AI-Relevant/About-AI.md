## AI相关概念体系

### 1.大语言模型（LLM——Large Language Model）
- **架构**：基于Transform
- **工作原理**：文字接龙游戏，预测下一个出现概率最高的词并基于这个词继续进行接下来的推测，直至结束。

### 2.Token
- **大模型本质**：数学函数，接收数字并输出数字
- **Tokenizer**：编码（文字->数字），解码（数字->文字）
- **概念**：在解码过程中将自然语言分成的最小单元
- **特性**：每一个Token映射成一个Token ID（数字）
- **注意**：Token != 词  ，无论是中文还是英文
### 3.上下文：Context
- **定义**：大模型每次处理任务时所接受到的信息总和
- **内容**：对话历史，用户问题，当前输出，工具列表，System Prompt
  
### 4.上下文窗口：Context Window
- **定义**: Context 能容纳的最大Token数量
  
### 5.提示词：Prompt
#### 5.1用户提示词（User Prompt）
- **定义**：用户发送给AI模型的消息，通常是提出的问题或表达的意图。
- **局限性**：
  - AI 模型扮演的是一个“大众”的“全才”。
  - 只能通过简单、中阶策略配合，缺少个性化的深度响应。
  - 缺乏实时数据和背景，仅仅依赖模型训练阶段及有限的上下文来生成回答。
#### 5.2 系统提示词（System Prompt）
- **产生背景**：为解决AI模型缺乏人设的问题，早期尝试将人设与用户请求打包进同一条用户提示词中，但这使得请求内容显得不自然。
- **定义**：将AI模型的人设、性格、背景信息、预期等独立出来的提示信息，只要不是用户直接说出来的内容，都可以置于系统提示词中。
- **机制**：每次用户发送用户提示词时，系统会自动将系统提示词一同发给AI模型，使得对话更自然。
- **应用场景**：
  - 用于创建具有特定性格、规则或特定领域知识的客服机器人、写作助手等。
  - 例如：在 ChatGPT 或 Claude 中，开发者通过 System Prompt 设定一个“资深代码评审专家”的角色。
  
### 6.AI Agent
- **通俗解释**：AI 模型被赋予了使用外部工具、完成复杂任务的能力，从而由“全才”变为“专才”。
- **概念**：
  - AI Agent：是一个以 AI 模型为核心，能自主思考、调用工具并执行任务的智能系统。
  - Agent Tool：是 Agent 可以调用的具体工具或 API。
- **特性**：
  - 不仅是文本输出：AI 不仅会给出回答，还会根据任务需求触发外部行为（例如发送邮件、查询实时数据库等）。
  - 循环反馈：用户或系统通过观察 Agent 运行的结果并给出反馈，促使 Agent 不断迭代计划。
  - 自主决策：AI Agent 会根据当前任务及环境，自主决定执行哪些步骤、选用哪些工具。

### 7.Agent Skills
- **本质**：提前写好并推给Agent的一份说明文档（markdown格式）
部分：
- **元数据层**：说明文档的封面，两个属性：name ，description
- **指令层**：描述要执行的事情
  

### 8.函数调用 (Function Calling)
#### 8.1 Function Calling 的概念
- **背景**：为解决 AI 模型在回答过程中无法直接执行外部功能的问题，模型厂商（如 OpenAI、Anthropic、Google）推出了 Function Calling 功能。
- 核心机制：把一组工具描述列表（JSON 格式）提供给 AI 模型供其随时调用。
#### 8.2 Function Calling 的机制
- **工具描述与交互**：
  - 包含五个主要元素：函数名称、描述、参数定义等。
  - 工具定义 (Schema)：模型收到任务及一组工具列表（JSON Schema），根据语义匹配决定调用哪个工具。
  - 工具调用：模型输出一段 JSON 格式的指令（指明要调用的函数和参数值），然后由程序在客户端执行该代码。
- **从输出到操作的跃迁**：Function Calling 并没有真的直接运行代码，而是生成一段结构化的数据，从而告诉了程序层面需要调用哪些外部函数。
- **优化场景**：
  - 解决了环境隔离带来的安全问题。
  - AI 模型只需负责逻辑推理与生成调用指令。
  - 由于大规模厂商针对性地优化了模型，其生成的指令质量非常高且具有强稳定性。
  - 因为是在协议层面的配合，对于复杂的场景，AI 模型能够同时调用多个工具并根据其返回的结果进行汇总。
#### 8.3 Function Calling 的局限性
- **绑定单一模型**：每个模型厂商的 API 定义不同，具有较强的平台依赖性。
- **开发成本高**：开发者需要为不同的模型重复编写这些工具。
- **效率问题**：Function Calling 仍然属于一种中心化的模式，对于大规模、跨部门、跨系统的协作仍显得笨拙。

### 9.MCP协议（Agent与Tool服务的桥梁）
#### 9.1 MCP 的诞生背景
- **问题**：当 AI Agent 与 Tool 服务的复杂度呈指数增长时（例如适配上百个模型、数百个不同的 API），每个 Agent 的接入成本极高。
- **解决方案**：借鉴 Agent Tool 的概念，提供统一的协议标准，即 MCP (Model Context Protocol)。
#### 9.2 MCP 的定义与组件
- **定义**：MCP 是一个开放式、标准化的协议，旨在简化 AI Agent 与 Tool 提供商之间的通信方式。
- **核心组件**：
  - MCP Server：管理并提供 AI Agent 可调用的工具资源。
  - MCP Client：集成 MCP Server 上提供工具功能的应用程序。
- **核心优势**：
  - 实现了 MCP Server 与 MCP Client 之间的解耦和复用。
  - 开发者只需实现一套符合协议标准的接口，各种模型即可直接接入并执行。
#### 9.3 MCP Server 提供的服务内容
- **Tool**：具体的函数调用和逻辑实现。
- **Resource**：静态内容，如文档、数据库内容的查询。
- **Prompt**：针对特定场景的预设提示词模板。
#### 9.4 MCP 的部署方式
- 可在 Agent 的本地环境中运行，也可以在远程服务器上分布式部署。
- 可采用标准的 HTTP 协议或更加高效的通信协议。
#### 9.5 MCP 与 AI 模型的关系
- **模型无关**：MCP 的每一个组件是标准化的，不依赖于任何特定的 AI 模型。
-** 桥梁**：MCP 是联结各种 AI Agent（底层大模型、框架、终端应用）和各种外部工具、数据资产的桥梁。
